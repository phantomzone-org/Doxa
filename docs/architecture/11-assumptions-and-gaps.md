# Assumptions & Gaps

| # | Item | Type | Details | Source |
|---|---|---|---|---|
| 1 | **Aggregated Input Proof leaf circuit is a placeholder** | Known gap | `aggregator_artifacts` builds the `GenericAggregator` around a trivial circuit with `n` unconstrained public inputs. The aggregation is cryptographically sound but the leaf proofs carry no real transaction-validity semantics until the actual `PrivateTx` circuit is defined and artifacts are regenerated against it. The full BN128 + Groth16 pipeline and `VerifierAggregator` are wired and functional. | `tessera-server/src/bin/aggregator_artifacts.rs`, `tessera-trees/src/proof_aggregation/generic.rs` |
| 2 | **Two-phase recovery drops all tx_proofs** | Known gap | `recover_pending_requests()` re-queues prove jobs using `DUMMY` sentinel proofs for all four trees because the original `tx_proof` bytes are not persisted across restarts. The prover expands all sentinels to the canonical all-zero padding proof and produces a mathematically valid `aggregated_input_solidity_proof`, but this proof carries no real transaction-validity guarantee. The on-chain verifier will accept it if the aggregation circuit verifying key matches. | `tessera-server/src/sequencer/recovery.rs` (`apply_and_requeue_pending_batch()`) |
| 3 | **Single-operator model** | Trust boundary | Only one `operator` address can submit proofs and manage the bridge. No multi-sig, rotation scheme, or decentralized operator set. | `tessera-solidity/src/TesseraRollup.sol` (`onlyOperator` modifier) |
| 4 | **No authentication on Sequencer API** | Security gap | All HTTP endpoints on `:8081` are unauthenticated. Anyone with network access can submit consume requests or private transactions. When `TESSERA_AGGREGATOR_ARTIFACTS_PATH` is configured, `input_proof` / `tx_proof` bytes are cryptographically verified against the leaf circuit at intake (`LeafProofVerifier`), which prevents malformed proofs from reaching the prover — but the endpoint itself remains open to all callers. | `tessera-server/src/sequencer/api.rs` |
| 5 | **Prover is a global FFI singleton** | Concurrency limit | `Groth16Wrapper` calls into Go via FFI with global state. `Arc<Mutex<ProverRuntime>>` serializes all proof requests. Only one proof can be generated at a time. Each `prove_request` call switches the active circuit twice: once for the tree proof (Commitment or Nullifier) and once for the aggregation root proof. Each switch requires reloading Groth16 keys from disk. | `tessera-server/src/prover.rs`, `tessera-trees/src/groth/wrapper.rs` |
| 6 | **Batch priority is hardcoded (deposit-only path)** | Design decision | For the deposit-only path (`/consume-request`), notes commitment always takes priority over other trees. Under sustained load, accounts trees may starve. The two-phase private-TX path bypasses this entirely — all 4 trees are registered atomically. No configurable priority for the deposit-only path. | `tessera-server/src/sequencer/pipeline.rs` (`maybe_start_next_batch()`) |
| 7 | **`insecure-stub-proof-verify` feature** | Cargo feature flag | Present in `tessera-server/Cargo.toml`. Likely bypasses proof verification for testing. Must never be enabled in production. Usage not fully traced. | `tessera-server/Cargo.toml` |
| 8 | **Tree depth fixed at 32** | Invariant | Hardcoded in commitment tree initialization. Supports a maximum of ~4 billion leaves. Not configurable at runtime. | `tessera-trees/src/tree/commitment_tree/tree.rs` |
| 9 | **No mempool / TX management** | Gap | Sequencer submits on-chain TXs but only polls for receipt with a 60s timeout. No nonce management, TX replacement (speed-up), or gas price adjustment is visible in the code. | `tessera-server/src/sequencer/pipeline.rs` |
| 10 | **Edition 2024 in tessera-trees** | Uncertain | `Cargo.toml` lists `edition = "2024"`. Standard Rust editions are 2015, 2018, 2021. This may be a nightly/unstable edition or a typo. | `tessera-trees/Cargo.toml` |
| 11 | **No rate limiting on API** | Gap | The sequencer API has no rate limiting. Channel capacity (1024) is the only backpressure mechanism. | `tessera-server/src/sequencer/api.rs`, `tessera-server/src/sequencer/mod.rs` |
| 12 | **Deposit commitment not enforced in nullifier path** | Design note | Notes submitted to `/notes/nullifier` are validated as `Validated` on-chain, but the nullifier value itself is not cryptographically bound to the original deposit. This relies on the (currently stubbed) transaction proof for correctness. The two-phase path (`/private-tx`) inherits the same constraint. | `tessera-server/src/sequencer/mod.rs` |
| 13 | **Two-phase batch queue silently drops on overflow** | Operational gap | When `registered_pending_batches.len() >= MAX_PENDING_BATCHES (128)`, new `/private-tx` requests are logged and dropped without returning an error to the client. The API returns `HTTP 503` only if the `private_tx_tx` channel is closed (service unavailable), not on queue overflow. On-chain, `pendingBatches` is a `mapping(uint256 => PendingBatch)` indexed by `batchId % MAX_PENDING_BATCHES`; the `_pendingCount` guard enforces the 128-slot limit. A `SlotConflict` revert is a defensive fallback if `_pendingCount` ever diverges from actual slot occupancy. | `tessera-server/src/sequencer/pipeline.rs` (`register_tx_batch()`), `tessera-solidity/src/TesseraRollup.sol` |
